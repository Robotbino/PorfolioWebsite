02.1The concept — star-map wayfinding

The cosmos is the navigation system, not decoration. The site is one continuous, downward-looping vertical scroll through five Destinations — Home, Experience (Work), About, Certifications, Contact. Each destination owns a real constellation; scrolling between destinations morphs one figure into the next, and passing the end wraps invisibly back to the start.

  • No Angular Router. The "pages" under src/app/pages/ are full-viewport <section>s stacked in one template. Navigation is scrolling (or nav-link travel), not route changes.
  • Immersion outranks scan-speed. The optimisation target is a visitor who engages with the experience, per CONTEXT.md. Motion is restrained: opacity, blur, long --ease-out-expo curves — never spectacle.
  • Accessibility is the baseline. The labeled nav always works (keyboard, screen reader), the loop clone is inert, and everything animated respects prefers-reduced-motion.

02.3The scroll loop (the spine of the site)

Vocabulary first (all from CONTEXT.md):

Cycle

One full pass Home → Work → About → Certifications → Contact, measured in "destination units" (0 = Home, 1 = Work, …).

Loop clone

A second, non-interactive copy of Home placed after Contact — inert + aria-hidden. It gives the Contact→Home morph its scroll distance and acts as the seam buffer.

Seam

The wrap point. When you reach the clone's top, the content on screen is pixel-identical to real Home, so subtracting exactly one cycle from scrollY is invisible.

One-direction

Only wraps downward. Scrolling up is normal native scroll and clamps at Home (bidirectional looping was rejected — it fights native scroll).

The mechanics, end to end:

  1. AppComponent measures every section's offsetTop (a ResizeObserver re-measures on any reflow) and hands the list to ScrollLoopService.setAnchors(). The last anchor is the clone — its offset is the cycle length in pixels.
  2. On every scroll event the shell calls loop.wrapOffset(scrollY). When the answer is non-null, the seam is due: the shell does window.scrollTo(0, adjustment), preserving momentum overshoot instead of snapping to 0.
  3. The shell also feeds loop.update(scrollY, innerHeight), which recomputes the continuous position signal via the pure functions in scroll-loop.math.ts.
Division of labour (ADR-0007)The shell owns all DOM I/O (reads and the scrollTo); ScrollLoopService owns the state; scroll-loop.math.ts owns the arithmetic and is fully unit-tested. Keep new logic on the correct side of that seam.

02.4Signals & the 60 fps hot path

The whole animation system runs on one requestAnimationFrame loop — FramePulseService — registered outside Angular's zone. Subscribers (the constellation morph, the nav mute) write straight to the DOM. Two signal rules keep change detection out of the hot path:

  • ScrollLoopService.position is hot — never bind it in a template. It is read ~60×/s from rAF callbacks as a plain position() call, which is non-reactive and schedules nothing. Binding it would reintroduce 60 fps change detection.
  • activeDestination is the safe one. A computed that collapses a whole section of positions into one id ('dest-work'…), so it notifies roughly once per destination. The nav binds this for the active-link underline and aria-current.

FramePulseService.onTick(fn) returns an unsubscribe function; the loop starts with the first subscriber and stops with the last. It also pauses on a hidden tab and resets its clock on return, so the first frame back never receives the whole hidden gap as dt.

02.5Theming model

Every colour token in src/styles.css is declared once using light-dark(<light>, <dark>), and color-scheme picks the side:

  • No class on :root → color-scheme: light dark follows the OS preference.
  • ThemeService (signals, persisted to localStorage, plus a pre-paint guard in index.html) adds .light-mode / .dark-mode, which only need to flip the scheme — no token overrides, with one exception (--scrim-rgb, a raw RGB triple that can't use light-dark()).
  • The aurora is a theme decision too: aurora-palette.ts maps dark/light to shader tuning, exposed as a memoized computed on ThemeService and bound to <app-aurora>'s inputs in the shell template.

03.2Routing logic — there is no router

Deliberately, RouterModule is not imported anywhere. The five "pages" are sections of one template, and "navigation" is scroll travel:

  1. The registry drives everything. destinations.ts exports the ordered DESTINATIONS array (id, label, figure). The nav renders its links from it and constellation.figures.ts derives its morph order from it — so ids can't drift.
  2. The shell template lays out the sections (app.component.html): five <section #dest id="dest-…" class="dest"> elements plus the inert Home clone at the end. Home is pre-revealed (loop anchor + LCP); Work/About/Certifications/Contact fade in via InViewportService.
  3. Nav clicks are travel, not navigation. Short hops smooth-scroll. Long jumps teleport instantly inside document.startViewTransition() (NavTransitionService), styled in styles.css as a directional drift-fade via data-nav-transition="up|down" on <html>.
  4. Two hand-written exceptions (by design, documented in destinations.ts): the #projects sub-anchor inside Work, and the desktop bar omitting Home because the logo is Home.
  5. Deep links: Netlify's /* → /index.html redirect means any URL serves the app.

04.3Tweak the constellation background

Everything lives in src/app/constellation/constellation.figures.ts. The one hard rule: every figure has exactly STAR_COUNT = 8 stars, because the morph interpolates star i of one figure to star i of the next, 1:1 by index. Specs enforce this — npm test after any change.

Move or reshape a figure

Each figure is index-ordered [x, y] coordinates in a 300 × 300 viewBox, plus its own line list:

// constellation.figures.ts — e.g. the Work figure (Orion)
work: figure('Orion', [
  [96, 96],    // star 0 — Betelgeuse   ← move a star by editing its [x, y]
  [176, 84],   // star 1 — Bellatrix
  // … exactly 8 coordinate pairs, always
], [
  { a: 2, b: 0 },  // a line between stars 2 and 0 (indices into the array above)
  { a: 3, b: 4 },  // link counts MAY differ per figure — lines cross-fade at the
]),                //   dim point of each morph, so they never need to match up

Change star character (size & drift)

Per-index traits are shared across all figures so star i keeps its identity through every morph:

// shared per-index traits — index i applies to star i of EVERY figure
export const R   = [3.3, 2.6, 2.2, 2.5, 2.7, 2.7, 2.6, 2.6]; // star radius
export const AMP = [6, 5, 5, 6, 6, 6, 7, 7];       // drift radius (viewBox units)
export const PHX = [0.0, 0.8, 1.6, 2.4, 3.2, 4.0, 4.8, 5.6]; // x-drift phase
export const PHY = [1.0, 2.2, 0.4, 1.7, 2.6, 0.9, 1.5, 3.0]; // y-drift phase
  • Bigger stars: raise values in R. Lazier/livelier drift: adjust AMP; phases (PHX/PHY) just de-synchronise the wander.
  • Morph behaviour (timing, dim point, interpolation) lives in constellation-morph.ts / morph-driver.ts — both spec-covered.
  • Reduced motion: the drift and morph settle automatically via MotionSettingsService; you don't need to handle it per figure.

04.4Tune the aurora

The WebGL aurora's look is a theme decision, so its knobs live in src/app/core/aurora-palette.ts — not in the shader or the shell template:

// src/app/core/aurora-palette.ts
export function auroraPalette(isDark: boolean): AuroraPalette {
  return isDark
    ? { colorStops: ['#1E1C1A', '#3D312A', '#5A3D2B'], blend: 0.5, amplitude: 1.0,  speed: 0.5 }
    : { colorStops: ['#FDFCFB', '#F5F2EE', '#FAF0E8'], blend: 1.0, amplitude: 0.25, speed: 0.5 };
}
KnobEffect
colorStopsThree-stop gradient the simplex noise flows through (left → right)
blendHow softly the aurora blends into the page background (higher = softer)
amplitudeVertical intensity of the waves — light mode keeps this low on purpose
speedFlow speed of the noise field

The shader itself (GLSL simplex noise via OGL) is in src/app/aurora/aurora.component.ts. Change it only if a palette knob can't express what you need — and keep the CSS fallback path working.

04.5Update design tokens

CSS custom properties, not SCSSThis repo has no SCSS. All "variables" are CSS custom properties defined once in src/styles.css under :root — typography scale, Nohemi weight ladder, tracking, colours, spacing, radii, shadows, easings, layout widths.

The theming rule: declare each colour once with light-dark(), never as separate light/dark overrides:

/* src/styles.css — the pattern for every colour token */
:root {
  color-scheme: light dark;                /* follows the OS with no class */
  --color-background:  light-dark(#ffffff, #121212);
  --color-accent-vivid: light-dark(#d9480f, #ff8c42);  /* the ONE non-monochrome hue */
  --ease-out-expo: cubic-bezier(0.16, 1, 0.3, 1);      /* the house easing curve */
  --font-size-display: clamp(3.25rem, 11.5vw, 11.5rem);
}
.light-mode { color-scheme: light; }       /* manual toggle only flips the scheme */
.dark-mode  { color-scheme: dark; --scrim-rgb: 18, 18, 18; }
  • Never hardcode hex values in component CSS — add a token in styles.css and reference it, or both themes break.
  • Don't add overrides to .light-mode/.dark-mode — they exist only to flip color-scheme (plus --scrim-rgb, the documented exception).
  • Motion tokens matter: new animation should use --ease-out-expo / --transition-slow — long and calm is the house style ("less but better").
  • Glass-card surface tokens (--glass-*) are in the same file; see ADR-0006 before touching them.

04.6Add a whole new destination

Rarer, but this is the full checklist. The registry keeps the nav and constellation in sync automatically; the shell template and module are the two manual touch points.

  1. src/app/destinations.ts — extend the DestinationFigure union and add the entry at its scroll position (order = array position): 
    export type DestinationFigure = 'home' | 'work' | 'about' | 'certifications' | 'contact' | 'lab';

{ id: 'dest-lab', label: 'Lab', figure: 'lab' },
  2. constellation.figures.ts — add a lab: figure to byRoute with exactly 8 stars (the destinations.spec.ts guard fails the build-time tests if the key doesn't resolve).
  3. app.component.html — add the section in scroll order, before the Home clone (the clone must stay last — it is the seam): 
    <section #dest id="dest-lab" class="dest"><app-lab></app-lab></section>
  4. Create the component under src/app/pages/lab/ and declare it in app.module.ts. Use <app-display-heading> for the section header and put its copy in a lab-data.ts, matching the house pattern.
  5. Verify: npm test (registry + figure guards), then scroll a full loop in the browser — check the new morph both in and out, the nav underline, and the seam still wrapping invisibly.
Do not touch the cloneThe final <section> (Home clone, inert + aria-hidden, no id) must remain the last child and pixel-identical to real Home — the invisible seam wrap depends on it. See ADR-0004.
