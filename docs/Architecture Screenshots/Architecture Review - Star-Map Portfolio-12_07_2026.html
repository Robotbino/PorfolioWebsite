The Star-Map Portfolio,
weighed module by module

Angular 19, one NgModule, one looping scroll page. A correction to the brief up front: there is no 2D-canvas star-field. The only pixel renderer is the OGL/WebGL aurora backdrop; the "stars" are an SVG constellation mutated per frame. The deep-module pattern you asked for already exists in embryo — FramePulseService, ScrollLoopService + pure *.math.ts, all unit-tested. The entropy lives in what surrounds them: duplicated cross-cutting decisions, three parallel notions of "current section", and tuning data smeared across templates, TS and CSS.

deep = small interface, big implementation shallow = interface ≈ implementation seam = where behaviour can be swapped deletion test = delete it; does complexity vanish?

4 · Already deep — extend, don't touch

ScrollLoopService + scroll-loop.math — the Cycle behind a 5-member interface; DOM I/O stays in the shell, math is pure and spec'd. This is the house pattern (ADR-0007); C3 extends it rather than replacing it.
FramePulseService — one rAF, out of zone, ref-counted, one-method interface. Every renderer already adapts to it; C4 deepens it with visibility handling.
MorphDriver + constellation-morph + figures — morph physics and interpolation as pure, spec-covered modules; the component is a thin SVG adapter.
certifications.math — tilt, mask, FLIP arithmetic extracted and tested; the 409-line component is heavy, but its math already sits behind a seam.
Aurora's @Input seam — the only true pixel renderer has the cleanest interface in the app (4 inputs hiding 300 lines + GLSL). C6 fixes what feeds it, not the module.
Doc drift note — README / CONTEXT / ADR-0007 still describe a 4-destination cycle; the code runs 5 (Certifications). Fold a docs-true-up into whichever candidate lands first.
